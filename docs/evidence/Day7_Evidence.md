## 6. AI 协作复盘

### 6.1 AI 在哪些阶段帮助最大

1. **项目初始化阶段（Day1—Day2）**：批量创建文件（页面占位、路由配置、API 模块），效率提升约 10 倍
2. **数据建模阶段（Day3）**：为四类业务场景设计 30 条贴切校园生活的 Mock 数据，数据结构设计合理
3. **代码审查阶段（Day4）**：四层交叉验证（db.json → Interface → buildPayload → 列表渲染），发现了人工容易忽略的字段不一致问题
4. **批量改造阶段（Day5—Day6）**：创建 Pinia Store、注册登录流程、全局联动改造，AI 能保持多文件修改的一致性
5. **代码清理阶段（Day6）**：并行扫描未使用文件/变量/API，发现 15+ 项冗余，比人工逐个检查高效
6. **文档整理阶段（Day7）**：辅助生成 README 结构、证据卡框架、展示提纲

### 6.2 AI 生成内容中出现过的问题

| 问题类型 | 具体表现 | 出现阶段 |
|---------|---------|---------|
| 过度设计 | ItemCard 组件使用插槽架构，但四个业务页面实际视觉差异大，组件未被使用 | Day3 |
| 保守保留 | 新旧 API 文件并存（trade.ts + secondHand.ts），未主动清理冗余 | Day3—Day6 |
| 交互粗糙 | 批量管理的 checkbox 默认一直显示，需人体验后指出才改为可选 | Day5 |
| 空值安全盲区 | `currentUser` 变为 nullable 后未全局检查空值访问，导致白屏 bug | Day6 |
| 过度建议 | 建议添加 JWT token、路由守卫、密码加密等超出范围的功能 | Day6 |
| 类型转换粗糙 | `as Record<string, unknown>` 对 ErrandItem 类型不兼容，需改为双重 as | Day7 |
| 模板类型推断 | DashboardView 中 `typeof allData` 捕获了 Ref 包装类型而非内部数据类型 | Day7 |

### 6.3 自己如何判断和修改 AI 生成内容

1. **功能边界判断**：拒绝了 AI 建议的 JWT token、路由守卫和密码加密——这些是真实项目的正确做法，但在本教学项目中超出了范围
2. **设计取舍判断**：ItemCard 保留但不强制统一——"先让每个场景做到最好，再寻找共性抽象"的务实思路
3. **交互体验判断**：批量管理模式从"checkbox 一直可见"改为"点击多选后才出现"，这个改进来自人工体验反馈
4. **代码质量判断**：类型安全问题是人工通过构建命令发现的，AI 不会主动做"下游影响分析"
5. **文档真实性的把关**：确保 README 和证据卡不虚构未实现的功能（如实说明无真实登录、无图片上传、无后端）

### 6.4 哪些内容必须由自己理解后才能完成

- 校园业务场景的理解和页面拆解（四类业务的差异化需求）
- 状态边界的判断（哪些数据放入 Pinia Store，哪些留在组件内）
- AI 过度设计 vs 合适设计的判断（什么阶段需要抽象，什么阶段不需要）
- 构建错误的根因分析和修复方向（TypeScript 类型系统的工作原理）
- 项目展示的表达逻辑（围绕"业务功能、技术实现、开发过程、问题解决、AI 协作"展开）
- 证据卡和复盘内容的真实性审核（AI 只能生成框架，真实体会必须是自己的）

### 6.5 以后使用 AI Coding 工具时会注意什么

1. **始终先跑构建验证**：AI 生成的代码通过 TypeScript 类型检查才算合格
2. **建立"人体验 → 反馈 → AI 修复"的循环**：不要期待 AI 一次性做到完美，交互细节需要人体验后反馈
3. **警惕 AI 的"保守保留"倾向**：定期主动检查冗余代码和未使用文件
4. **在关键决策点暂停，自己判断后再让 AI 执行**：不要盲从 AI 的建议
5. **使用"扫描-判断-执行"三步模式**：让 AI 扫描问题 → 人来判断哪些改哪些不改 → AI 精准执行
6. **文档和证据卡必须人工审核**：AI 可能虚构功能或遗漏重要细节

## 8. 项目不足与改进方向

### 当前存在的不足

| 不足 | 说明 |
|------|------|

| 模拟登录 | 用户登录为前端 localStorage 模拟，无真实认证和会话管理 |

| 无后端校验 | 内容违规检测仅在前端，可被绕过 |

## 9. 今日反思

通过 7 天实训，我对前端项目开发、AI 协作开发和工程化过程有了以下新的理解：

**关于前端工程化**：一个完整的前端项目不只是"写页面"，而是一个从需求理解、页面设计、数据建模、接口模拟、状态管理、交互优化到项目展示的完整过程。7 天中经历了从零搭建到项目验收的全流程，每个阶段都有其特定的技术决策点——比如 Day3 时 API 模块是集中式还是按业务拆分、Day5 时状态管理的边界在哪里、Day6 时代码清理的优先级。

**关于 AI 协作开发**：最大的收获是理解了"AI 是加速器，不是替代者"。AI 在批量生成、结构化审查、代码清理方面效率极高，但它有三个明显的盲区：（1）不擅长评估"类型变更的下游影响"；（2）倾向于过度设计或保守保留两种极端；（3）缺乏"阶段感"——不知道在项目的什么阶段应该做什么程度的抽象。这就要求开发者不仅需要 coding 能力，更需要业务判断力、架构决策力和质量把关意识。

**关于综合验收**：Day7 让我认识到，项目的"完成"不止于代码能运行。它还包括：功能是否形成闭环、结构是否清晰、文档是否完整、过程是否可追溯、问题是否被记录、AI 协作是否有反思、最终成果是否能被清楚展示。这些"非代码"的工作同样是软件工程的重要组成部分。

**关于过程证据**：7 天证据卡和 Git 提交记录形成了项目开发的完整链条。回头看，每天的记录不仅是"证明做了"，更是一种学习沉淀——在写的过程中梳理了当天的技术决策、AI 协作经验和遇到的坑。这种"做中学、写中思"的方式对后续项目也有借鉴意义。

**综合验收、项目展示、AI 复盘**三个环节构成了 Day7 的核心价值：验收保证了项目的工程质量，展示训练了技术表达能力，复盘沉淀了 AI 协作的方法论。三者缺一不可。
