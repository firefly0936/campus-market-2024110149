## 四、开发重点

1. **组件化思维** — 四类业务虽有差异，但列表/详情/发布模式相似，应抽取通用组件（如 `ItemCard`、`ItemForm`），减少重复代码。

2. **数据流管理** — Pinia Store 负责跨页面数据共享（当前用户、列表数据），API 层统一管理后端请求，视图层只关注渲染。

3. **AI 协作规范** — 每次 AI 交互记录 Prompt / 输出 / 修改 / 验证，保持 AI_Collaboration_Card 同步更新，这是课程评价的重要依据。

4. **Git 提交习惯** — 每完成一个独立功能提交一次，提交信息遵循约定式提交（`feat:` / `fix:` / `docs:` / `chore:`）。

5. **过程性证据** — 每天结束后及时填写对应 Day 的 Evidence，真实记录学习过程，而非事后补写。

## 五、技术风险与应对

| 风险 | 应对 |
|------|------|
| JSON Server 数据不会持久化 | db.json 纳入 Git 版本管理，初始数据始终可用 |
| ECharts 配置复杂 | Day6 预留充足时间，先用简单图表验证再美化 |
| 四类业务逻辑相似导致代码重复 | 提前抽取通用组件和 composables |
| AI 输出质量不稳定 | 人工审查所有 AI 生成代码，保持最终决策权 |
