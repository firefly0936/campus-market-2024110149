## 一、项目需求理解

### 1.1 项目定位

校园轻集市是面向高校学生的轻量级校园生活服务平台，本课程是 **AI 辅助下的前端工程实践项目**，重点在于 Vue3 工程化开发与 AI Coding 协作，而非完整产品交付。

### 1.2 四类业务场景分析

| 场景 | 核心功能 | 关键数据 |
|------|----------|----------|
| 二手交易 | 发布/浏览/搜索二手商品 | 价格、成色、分类、图片 |
| 失物招领 | 发布失物/招领信息 | 类型（失物/招领）、地点、状态 |
| 拼单搭子 | 发起拼单、参与拼单 | 目标人数、当前人数、截止时间 |
| 跑腿委托 | 发布委托、接单 | 报酬、取件地、送达地、状态 |

### 1.3 用户故事与功能边界

- 用户需先创建校园身份，然后可发布和浏览四类集市信息
- 本课程聚焦前端实现，后端使用 JSON Server 模拟
- 不涉及真实支付、实名认证、即时通讯等复杂功能

## 二、技术架构理解

| 技术 | 作用 |
|------|------|
| Vue3 + TypeScript | 前端框架，组件化开发 |
| Vite | 构建工具，开发服务器 |
| Vue Router | 前端路由，页面导航 |
| Pinia | 状态管理，跨组件数据共享 |
| Axios | HTTP 客户端，调用后端 API |
| JSON Server | Mock 后端，基于 db.json 提供 RESTful API |
| ECharts | 数据可视化，趋势看板图表 |
| Git | 版本管理，代码提交与协作 |
| AI Coding 工具 | 辅助开发，代码生成与审查 |

### JSON Server 使用原因

1. 前端实践项目无需搭建真实后端
2. 基于 JSON 文件，零配置即可提供 RESTful API
3. 支持 CRUD 完整操作，模拟真实前后端交互
4. 学生可专注于 Vue3 前端开发技能

## 八、实验思考

### 8.1 为什么课程没有要求第一天立即开发页面？

第一天的任务是"理解项目"而非"编码实现"，原因有三：

1. **方向比速度重要。** 如果不理解四类业务场景的差异（二手交易需要价格/成色，拼单需要目标人数/截止时间），直接写代码会导致数据结构反复返工。

2. **基础设施决定上限。** Day1 搭建的路由懒加载、API 层封装、目录分层、TypeScript 配置，这些不会直接产生可见的页面，但后续六天都在这个骨架上盖楼。骨架歪了，楼就歪了。

3. **工程习惯的培养。** 真实项目中，需求评审、技术方案、目录规范都是编码前必须完成的步骤。Day1 模拟的就是这个"动手前的准备"阶段。

### 8.2 软件工程中"理解项目"为什么比"开始编码"更重要？

- **理解错了，代码白写。** 如果把"拼单搭子"理解成电商拼团，加了支付逻辑，就超出了课程边界，白白浪费时间。
- **理解透了，代码自然快。** 今天把四类业务的数据结构梳理清楚后，创建 db.json 和 API 封装一气呵成，因为脑中已经有了清晰的模型。
- **理解是团队协作的前提。** 如果每个人对"集市列表页应该展示什么字段"理解不同，合并代码时必然冲突。

**核心结论：编码是理解的翻译过程，理解到位了，编码只是体力活。**

### 8.3 AI 在软件开发中应该承担什么角色？

AI 应承担 **加速器** 和 **副驾驶** 的角色，而非**替代者**。

| 维度 | AI 适合做的 | 人必须做的 |
|------|------------|-----------|
| 文件创建 | 批量生成样板代码、配置文件 | 审查生成结果是否合理 |
| 信息检索 | 分析项目结构、解释技术概念 | 判断信息准确性和适用性 |
| 重复劳动 | 安装依赖、运行检查命令 | 决定选型和架构 |
| 文档撰写 | 按模板填充结构化内容 | 确保内容真实反映学习过程 |
| 业务决策 | 提供选项和参考方案 | 做出最终选择并承担后果 |

今天的实践验证了这一点：AI 20 分钟完成了 20 个文件的创建和配置，但 db.json 字段设计是否合理、路由命名是否清晰、文档是否真实记录了学习过程——这些判断全部由人完成。

**核心原则：AI 出力气，人做判断。**

### 8.4 今天最大的收获

**项目初始化不是"创建项目"这一个动作，而是一整套工程基础设施的搭建。**

具体来说：
1. 一个能跑的项目 ≠ 一个工程化的项目。TypeScript 类型检查、ESLint 规范、目录分层、路由懒加载、API 层封装——这些第一天不搭好，后面越写越乱。
2. 文档先行比代码先行更重要。`Project_Plan.md` 把七天要做的事提前想清楚，比闷头写代码效率高。
3. Git 提交不是应付检查，而是工程习惯。每完成一个独立任务提交一次，提交记录就是最清晰的开发日志。
4. AI 可以大幅提效，但前提是人知道自己要什么。如果自己对项目结构都没概念，AI 生成什么就只能接受什么，失去了判断力。
