# Day4 Evidence — 发布表单与数据新增

## 1. 今日完成内容

今天完成了从表单设计到数据提交的完整链路：

- **发布表单**（`PublishView.vue`）：四类业务统一发布入口，通过顶部类型选择器切换表单内容，每个类型拥有独立的专属字段区域
- **表单校验**：为四类发布类型分别编写了校验规则，覆盖必填校验、数值范围校验、类型约束校验
- **数据新增**：通过 Axios POST 请求将表单数据写入 JSON Server，成功后自动跳转到对应业务列表页
- **内容违规检测**：创建了 `contentFilter.ts`（50+ 违规关键词 + 4 种正则模式）和 `ViolationDialog.vue` 弹窗组件，在表单提交前进行内容审核
- **通用组件**：创建了 `FormField.vue`（表单字段容器组件，统一 label + required + error 展示）
- **详情页**（`DetailView.vue`）：支持四类业务的详情展示，含面包屑导航、联系方式查看、三态处理

## 2. 发布类型与字段设计

| 发布类型 | 对应数据集合 | 关键字段 | 设计理由 |
|---|---|---|---|
| 二手交易 | secondHand | title、category、price、condition、tradeLocation、description | 二手交易的核心是"物品+价格+成色"。category 支持分类筛选（电子产品/教材教辅/出行工具/生活用品），price 支持排序，condition 是买家信任基础，tradeLocation 方便约定线下交易地点 |
| 失物招领 | lostAndFound | title、type、itemName、location、eventTime、description | type（失物/招领）是失物招领最核心的区分维度，决定页面徽章颜色（红/绿）；itemName 帮助快速识别物品；eventTime 帮助判断时间范围 |
| 拼单搭子 | groupBuy | title、groupType、targetCount、deadline、meetingLocation、description | targetCount 是成团门槛，配合 currentCount（提交时默认为 0）计算进度；deadline 制造紧迫感；groupType 区分拼单场景（外卖/团购/拼车/学习搭子/运动搭子） |
| 跑腿委托 | errand | title、taskType、reward、pickupLocation、deliveryLocation、expectedTime、description | pickupLocation 和 deliveryLocation 定义任务路线（取件地→送达地），reward 区分有偿/免费，expectedTime 是任务截止时间 |

**通用字段**（四类共有）：title（标题）、description（描述）、publisherId（发布者 ID，模拟为 1）、createdAt（创建时间）

## 3. 表单校验规则

### 通用校验
- **标题**：不能为空（`!form.title.trim()`），所有类型均适用
- **描述**：不能为空（`!form.description.trim()`），所有类型均适用
- **地点**：二手交易和失物招领类型下不能为空

### 专属校验

**二手交易：**
- 商品分类不能为空（必须选择一个分类）
- 价格不能为负数（`form.price < 0`）
- 成色不能为空（必须选择：全新/九成新/八成新/七成新及以下）

**失物招领：**
- 物品名称不能为空（`!form.itemName.trim()`）
- 时间不能为空（`!form.eventTime`）

**拼单搭子：**
- 拼单类型不能为空（必须选择：外卖拼单/团购/拼车/学习搭子/运动搭子/其他搭子）
- 目标人数不能少于 2 人（`form.targetCount < 2`）
- 截止日期不能为空

**跑腿委托：**
- 任务类型不能为空
- 酬劳不能为负数（`form.reward < 0`）
- 取件地点不能为空
- 送达地点不能为空
- 截止时间不能为空

### 校验触发时机
在 `handleSubmit()` 中，校验在构建 payload 和内容违规检测之前执行。校验失败时，对应字段下方显示红色错误提示，阻止表单提交。

1. **失物招领的 `itemName` 字段多余**：`buildPayload()` 提交了 `itemName`，但 `LostFoundItem` 接口和 db.json 中都没有这个字段，属于"孤儿数据"

## 5. 人工调整内容

对 AI 生成的内容做了以下修改和判断：

1. **itemName 字段**：确认该字段冗余，保留在 payload 中暂未删除——JSON Server 是 schema-less 的，额外字段不会导致错误，但已标记为后续清理项

2. **contact 字段**：确认缺失，但考虑到 Day4 任务要求"不要加入真实登录和复杂权限"，contact 涉及真实联系方式，决定暂时保留缺失状态。表单中的"发布须知"已有安全提示

3. **API 冗余**：确认 `trade.ts` 和 `secondHand.ts` 功能重叠，但此时不清理——两个文件分别被不同视图引用，贸然合并可能导致路由断裂，留待 Day7 重构

4. **price 歧义**：确认 0=免费的语义在校园场景可接受（闲置物品确实有免费送的情况），暂不修改为"未填写"和"免费"两个独立状态

5. **词库边界审查**：检查了 AI 生成的 50+ 个违规关键词，发现缺少"香烟""电子烟"等校园违禁品词条，已建议在实际使用中按需追加。"加微信"作为 warning 级别可能存在高误报率，需要持续调优

6. **正则误判评估**：银行卡号正则 `\b\d{16,19}\b` 可能误判订单号/快递单号，但考虑到其为 warning 级别（不阻止发布），误判成本可接受

7. **表单切换重置**：确认 AI 添加了 `watch(publishType, () => resetForm())`，确保切换发布类型时清空上一类型的残留字段——这个设计是正确的

8. **内容过滤定位确认**：明确了当前内容过滤是"善意提醒"而非"安全防线"——前端校验可被绕过，真正的安全需要后端再做一次校验

## 7. 遇到的问题与解决方法

**问题一：切换发布类型后字段残留**

在测试中发现，先选择"二手交易"填写了价格和成色，再切换到"失物招领"时，之前填写的价格和成色数据仍然保留在表单数据对象中。虽然模板中用 `v-if` 切换了专属字段的显示，但 `form.price` 和 `form.condition` 仍然有值，可能导致 `buildPayload()` 构建出混合了不同类型的异常数据。

**解决方法**：AI 添加了 `watch(publishType, () => resetForm())`，在发布类型切换时调用 `resetForm()` 清空所有表单字段并清除校验错误。同时为专属字段的 `<div>` 添加了 `<Transition name="fields-fade">` 动画，让字段切换视觉上更平滑。验证：切换类型后所有输入框恢复初始状态，不会再出现"失物招领表单里藏着二手交易价格"的情况。

**问题二：JSON Server 端口不一致导致 POST 请求失败**

项目中存在两个 axios 实例配置——`src/api/http.ts` 指向端口 3001，`src/api/index.ts` 指向端口 3000。`pnpm dev:server` 启动 JSON Server 在 3000 端口，而四个业务 API 文件全部导入 http.ts（3001）。如果只运行 `pnpm dev:server`，POST 请求会因为端口不匹配而失败。

**解决方法**：明确统一使用 `pnpm mock`（端口 3001）作为数据服务。开发时使用两个终端——终端 1：`pnpm mock`（JSON Server @ 3001），终端 2：`pnpm dev`（Vite @ 5173）。同时在 `http.ts` 的 baseURL 注释中标注了端口信息，避免后续混淆。

## 8. 今日反思

**发布表单**是用户向系统"输入数据"的唯一通道——没有发布表单，校园轻集市就只能浏览 Mock 数据，永远是一个"只读"的展示页，无法成为一个"活的"交易平台。表单的设计本质上是**业务建模的具象化**：二手交易需要 price 和 condition，拼单需要 targetCount 和 deadline，跑腿需要 pickupLocation 和 deliveryLocation——这些字段不是拍脑袋决定的，而是从四类业务场景的真实需求中推导出来的。

**表单校验**是前后端之间的第一道防线。如果说后端校验是"安全检查"，那前端校验就是"用户体验保障"——用户填错价格、忘记选分类，不应该等到点击发布按钮后服务器返回 400 才知道，而应该在离开输入框的那一刻就看到红色提示。今天实现的校验规则覆盖了四类业务的必填字段和数值范围，虽然简单，但已经让表单从"能填"变成了"好用"。

**数据新增**是 Web 应用从"展示型"走向"交互型"的分水岭。今天的 `createTrade()` → `http.post('/secondHand', data)` → JSON Server 写入 db.json 这条链路打通之后，校园轻集市就不再是"看看别人发布的东西"，而是"我也可以发布，别人可以看到我的发布"。这个从"消费者"到"生产者"的身份转变，是交互式 Web 应用的本质特征。

今天还有一个重要收获：**内容安全不是"加个功能"就能解决的事**。AI 帮我生成了 50+ 个违规关键词和 4 种正则模式，看起来覆盖面很广，但我审查时发现了两个关键问题：一是词库有盲区（"香烟""电子烟"未被收录），二是前端检测本质上是"纸门"——浏览器 DevTools 可以直接绕过。这让我认识到：内容过滤需要持续迭代（词库在实战中优化），而且前后端各司其职——前端做"用户体验层面的提醒"，后端做"数据层面的安全兜底"。

总结：**发布表单、表单校验和数据新增**三者构成了 Web 应用的"数据入口"——表单负责收集信息，校验负责保证信息质量，新增负责将信息持久化。少了任何一个环节，用户的数据要么进不来，要么进来的质量参差不齐，要么进来了却存不住。今天把这条链路打通了，校园轻集市才真正成为用户可以"参与"的平台。
